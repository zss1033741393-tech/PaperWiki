问题与背景

“自我改进”经常被想象成模型直接修改自己的参数，但现实中的 agent 是复合系统：模型之外还有提示词、工作流、工具、记忆、Skills、子 agent、评价器和运行时策略。即使参数 $\theta$ 完全不变，只要这些持久的外部状态 $H$ 发生改变，下一次运行的能力和行为就可能改变。

文章试图解决的不是“哪种方法最强”，而是如何用一个稳定框架阅读迅速分化的文献：提示词优化、工作流搜索、代码型 agent 演化、Skill 生成、Harness 自动优化和真正的自指更新究竟有什么共同结构，又应在哪些维度上严格区分。

论证结构

全文按“定义对象—建立坐标—分别考察组件—提出未解问题”推进：

  1. 第 1 章先把分析单位从模型扩大到部署中的 agent 系统，定义 Skill、Harness、自改进和自指。
  2. 第 2 章提出最小自改进闭环，并用“可编辑状态 × 更新者身份”两个主轴组织 agent 系统文献，再叠加证据、搜索拓扑、更新时间和参数耦合。
  3. 第 3 章把 Skill 视为可移植、可验证、可维护的外部能力包，讨论其发现、蒸馏、演化、部署和验证。
  4. 第 4 章把 Harness 视为更广的执行脚手架，沿更新对象、更新者、证据来源和更新时间分析六类系统。
  5. 第 5 章指出真正困难之处不是生成修改，而是在缺少真值时确定优化方向、证明改进可持续，并协调外部状态与模型参数的不同学习时间尺度。

逐章中文转述

1. 预备概念：分析的不是孤立模型，而是可持续修改的系统

1.1 范围、核心主张与分析单位

作者把部署中的 agent 写成 $A_t=(\theta_t,H_t)$：$\theta_t$ 是模型参数，$H_t$ 是持久的模型外状态。本文主要讨论 $\theta_{t+1}=\theta_t$、但 $H_{t+1}\neq H_t$ 的情形。普通微调不在主线内；只改善当前回答的一次性反思、best-of-$N$ 选择和临时上下文适应，也不自动构成自改进，因为它们没有留下能改变未来运行的持久状态。

完整系统还包括更新器 $U_t$、谱系或档案 $L_t$、评价与晋升机制 $V$，以及治理边界 $B$。这些角色在实现上可能交叠，但功能上必须区分。尤其当候选系统既能修改自身行为，又能修改评价标准时，分数上升不再足以证明能力真的提高。

作者据此提出中心命题：自改进是一个由证据驱动的持久状态转移，其对未来行为的影响必须可被评价。

[定位：H2 1.1，第 1—8 段及符号表]

1.2 Agent 系统是持久、可编辑的状态

外部状态可粗略分成三部分：声明式状态（提示词、规则、Skill 文档、配置）、程序式状态（工作流、工具代码、路由与重试逻辑）和经验状态（记忆、轨迹、失败案例、评价记录）。这些边界并不绝对，但能迫使分析者明确：到底哪一层被修改、修改能否跨运行保存。

同样需要区分“执行系统”和“改进系统”。前者完成任务，后者观察结果、提出候选修改、验证并决定是否保留。模型、Skill 或 Harness 都可能出现在两边；关键不是文件归属，而是功能角色和授权边界。

[定位：H2 1.2，全文]

1.3 Skill 是什么

本文采用严格的工件定义：Skill 是可复用、持久、可移植的能力包，概念上包括元数据与触发条件、根指令、脚本、参考资料或资产、测试与验证契约、依赖，以及来源和版本历史。不是每个 Skill 都必须有全部字段，但这个分解能说明什么可编辑、什么应被审计。

Skill 不只是提示词，因为它需要可发现、可复用和生命周期管理；也不等同于工具，因为它可以教 agent 在何时、为何以及如何组合多个工具；它更不是整个 Harness，因为它通常封装局部任务能力，而非控制全局运行时。

[定位：H2 1.3，第 1—6 段及 Skill 结构定义]

1.4 Harness 是什么

Harness 是位于模型与环境之间、控制 agent 如何运行的模型外执行脚手架，包括系统提示、上下文装配、工具与权限请求、工作流、子 agent、记忆、重试、验证、停止条件、日志和运行策略。

但 Harness 必须有边界：能力路由可以属于 Harness，凭证执行、权限上限、隐藏测试、不可变评分路径、sandbox、审计记录和最终回滚权应处于治理包络 $B$。否则候选 Harness 可能通过放宽测试或隐藏失败来“提高”自己的分数。

[定位：H2 1.4，全文及术语对照表]

1.5 自改进、演化和自指并非同义词

自动优化只说明存在机器生成的修改；演化强调跨候选、代际或档案的搜索；自指则要求更新者与被更新对象存在身份上的回环，例如当前 agent 参与构造自己的继任版本。一个固定优化器可以持续产生更好的 agent，但它不是自指；一个 agent 可以修改自己，却未必真的改善。

因此不存在从“低级”到“高级”的单轴阶梯。编辑范围、更新者身份、证据质量、搜索拓扑、更新时间、参数耦合和外循环可变性是彼此独立的维度。

[定位：H2 1.5，全文及术语表]

2. 自改进 Agent 系统：从最小闭环到自指演化

2.1 为什么需要自改进

部署行为由模型与外部脚手架共同决定。对闭源权重而言，外部状态甚至是应用开发者唯一现实的学习载体。一次临时洞见会在每个上下文窗口里重复付费；持久改进则可跨任务、用户和部署复用，并且比参数更容易检查、diff、测试、迁移和回滚。

长期而言，递归改进是通往更强智能的一条研究假设，但今天的 bounded agent loops 不能被直接等同于单调、无限的智能爆炸。更具体的问题是：哪些东西必须可编辑、可观察且可独立验证，改进才能可靠累积？

[定位：H2 2.1，第 1—7 段]

2.2 操作性定义与最小闭环

作者严格分开两个命题：

  • 自改进过程：执行、评价或环境证据促成持久的候选状态修改，目标是改善未来行为。
  • 已验证的改进：独立结果评价证明修改后的系统相对适当基线确实更好。

最小闭环需要四项条件：修改具有持久性；证据参与修改；存在把证据转换为有界候选差分的有效更新算子；后续行为与基线接受结果评价。其流程可概括为：运行当前系统 → 收集证据 → 诊断 → 提出候选差分 → 在边界内应用 → 独立评价 → 接受、拒绝或归档 → 进入下一轮。

由此得到三个重要区分：候选生成不等于接受；进入档案不等于上线；诊断反馈不等于最终裁决。在线修改可以先进入轨迹，但在获得后续证据前，准确说法应是“适应”而非“改进”。

[定位：H2 2.2，第 1—10 段及最小闭环图]

2.3 两个主轴与四个正交叠加层

第一主轴是可编辑集合 $\Omega\subseteq H$：提示词、工作流、工具、Skills、记忆、上下文与运行时中的哪些部分可以改变。工件使用文本、图、目录、配置还是代码，只是表示形式，不等同于语义上的更新对象。

第二主轴是更新者—目标关系：固定外部优化器修改目标；同模型但不同角色修改目标；历史版本生成继任者；可编辑的元 agent 同时修改任务 agent 和自身。

四个正交叠加层分别是：证据与评价类型；贪心、Pareto、MCTS、种群、开放档案等搜索拓扑；实验间、部署间或同一连续轨迹内的更新时间；冻结模型、交替更新或联合更新 $(\theta,H)$ 的参数耦合模式。

[定位：H2 2.3，第 1—8 段及系统坐标表]

2.4 固定优化器演化 Agent 工件

GEPA 在提示词层从完整轨迹和评价反馈中反思失败，并以 Pareto 选择保留互补候选。AFlow 把可编辑面扩展到由节点、操作符、边和提示组成的工作流，用 MCTS 搜索可运行流程。ADAS 再扩展到代码定义的目标 agent，由固定 Meta Agent 读取档案、生成与修复候选并评价。

三者可分析为编辑面的扩大，但这不是论文间已经声明的方法继承。共同边界是：被优化工件在变，设计它的优化器保持固定。

[定位：H2 2.4，第 1—6 段]

2.5 从 Agent 设计到自指演化

SICA 允许历史高效用 agent 分析版本档案并实现下一候选，效用同时考虑成绩、成本和时间。相较 ADAS，关键变化不是“工作流变成代码”，而是演化谱系中的 agent 开始参与构造继任者。

DGM 使用开放档案维护分支谱系，抽样父代、修改代码、评价后代；暂时较弱的分支也能保留为未来跳板。Hyperagents 更进一步，把 TaskAgent 与 MetaAgent 放进同一可编辑程序，后者读取评价并同时修改任务 agent 与自己，使“改进机制本身”也进入可编辑状态。

但自指仍不是无限自我批准。评价器、任务分布、资源限制、sandbox 和部分父代选择仍在外部，构成必要信任根。

[定位：H2 2.5，第 1—13 段]

2.6 超出 Agent 边界的程序演化

Learning Beyond Gradients 展示 coding agent 如何通过读取失败、编辑启发式策略、增加测试、检查日志与回放，让外部软件系统持续学习。AlphaEvolve 则以自动评价器为中心，从程序数据库采样并生成代码差分，用评分反馈驱动算法与基础设施程序演化。

判断边界的实用规则是：若被演化程序持续控制 agent 将来如何感知、行动、记忆或改进，它属于 agent/Harness 演化；若它只是外部任务解，则是相邻的程序演化。两者共享差分突变、种群管理、多目标评价和 evaluator-first 设计，但不应混为一类。

[定位：H2 2.6，第 1—8 段]

2.7 小结

第 2 章形成两条互动扩展线：一条扩大被编辑的系统表面，另一条让更新者与目标的关系越来越自指。Skill 与 Harness 不是这条“尺寸阶梯”的后续级别，而是按组件边界重新切分同一空间：Skill 强调可移植能力工件，Harness 强调整体执行脚手架。

[定位：H2 2.7，全文及关系证据表]

3. Skill 演化：改进外部化能力

3.1 可演化对象

Skill 把能力从每次上下文重新构造的临时知识，变成版本化、可发现、可测试、可回滚的持久包。它“像参数”一样改变未来行为，却不是参数：人仍可读它、迁移它、审计它。核心问题也因此从“如何写好说明”变成如何发现、构建、验证、维护和部署可复用能力。

[定位：H2 3.1，第 1—7 段]

3.2 Skill 从哪里来

EvoSkill 从带标签失败中诊断缺失能力，再生成或修改结构化 Skill，并以 held-out 表现保留 Pareto 候选。Trace2Skill 同时分析多个成功与失败轨迹，把局部经验合并成跨任务、跨模型可迁移的声明式目录，重点在“发现缺什么”和“怎样压缩重复经验”。

CoEvoSkills 面对只有不透明通过/失败信号的任务，让 Skill 生成器与信息隔离的替代验证器共同演化：外部 oracle 失败时，不仅修改 Skill，也修正验证器。OpenSkill 将证据扩展到开放网络资源，加入检索、来源追踪和外部验证，但同时引入恶意内容、许可、过期知识与供应链风险。

[定位：H2 3.2，第 1—12 段]

3.3 完整生命周期：优化、维护与部署

SkillOpt 关注已有 Skill 的执行策略优化，通过评价反馈改进调用与组合方式。MUSE-Autoskill 将创建、记忆、管理和评价组成持续循环，避免 Skill 只增不减。SkillClaw 汇总跨用户、跨时间的交互经验，把局部改进同步到共享库；这扩大了收益，也把错误局部编辑升级成供应链风险。

因此完整生命周期应包含发现、生成、单元验证、注册、检索、运行监控、修订、合并、弃用和回滚，而不是一次性生成文件。

[定位：H2 3.3，第 1—9 段]

3.4 何时可以声称变好

不同系统的验证强度差异很大：可使用任务标签、运行错误、单元测试、黑盒 oracle、替代验证器、用户历史、外部来源或执行反馈。作者强调至少要区分三层：语法或契约有效；在目标任务上有局部效果；跨任务、模型、用户与时间仍有净收益。

如果测试与 Skill 同时由同一更新器生成，就可能共同适配错误假设。稳健方案需要信息隔离、隐藏评价、回归集、成本约束、来源追踪、灰度发布和可回滚版本。

[定位：H2 3.4，全文及 Skill 系统对照表]

3.5 Skill 与 Harness 的边界

Skill 是快速、可检查的适应层，但优质 Skill 仍可能因为检索时机错误、上下文不足、工具路由不当或依赖失效而失败。Continual Harness 说明了这一点：它能在线修复单个导航 Skill，但整体更新对象是系统提示、子 agent、Skills 与记忆组成的复合运行时。Skill 演化常是 Harness 演化的一部分，不能只看文件夹名称判断。

[定位：H2 3.5，全文]

4. Harness 演化：改进执行脚手架

4.1 Harness 改进的坐标系

Harness 演化可沿六个坐标描述：可编辑面 $\Omega_H$；更新者 $U$；证据 $E$；搜索谱系 $L$；更新时间 $T$；参数耦合 $P_\theta$。评价 $V$ 与治理 $B$ 原则上应保持为受保护层。

这套坐标避免把所有“自动改 Harness”都当成同一种能力。窄范围代码过滤器、端到端 Harness 配置、自指修改、基于历史的离线更新、同一轨迹内在线更新，在安全性与证据要求上完全不同。

[定位：H2 4.1，全文及坐标定义]

4.2 更新什么

AutoHarness 生成用于约束环境交互的可执行代码，如合法动作过滤器、输出验证器和无需每步调用模型的代码策略。Meta-Harness 优化更宽的端到端模型脚手架。Agentic Harness Engineering 则以可观测性驱动 coding-agent Harness 的自动演化，把日志和结构化运行信号用于诊断、编辑和复测。

编辑面越大，潜在收益越高，归因与回归风险也越大。代码可编译并不意味着行为安全；一个通过局部测试的全局路由变化仍可能破坏其他任务。

[定位：H2 4.2，第 1—9 段]

4.3 谁执行更新

Self-Harness 把更新者坐标推向自指：同一基础模型在不同上下文和角色中运行，当前 Harness 的 agent 参与修改自身 Harness。它仍把评价器、隐藏划分、接受规则、权限上限和部署权放在可编辑状态之外。

因此“自己生成补丁”和“自己批准上线”必须分开。前者扩大提案能力，后者会破坏改进声明的可信度。

[定位：H2 4.3，全文]

4.4 从无标签历史中学习

Retrospective Harness Optimization（RHO）从过去任务中选择困难且多样的子集，多次重跑并分析分组轨迹。自验证找出可疑工具调用、假设与停止决定；自一致性暴露关键分歧；多个候选修改再通过成对自偏好比较。若没有候选相对基线获得正的平均偏好，系统保持不变。

这说明没有人工标签时仍能构造诊断证据，但自偏好只能作为搜索信号，不能自然升级为高风险部署的唯一权威。

[定位：H2 4.4，全文]

4.5 无重置在线适应

Continual Harness 在一条持续轨迹内允许 Refiner 原位修改系统提示、子 agent、Skills 和记忆。失败、修复和复用发生在同一环境状态中，突出更新时间 $T$ 的变化。

在线更新的难点是因果混杂：Harness 改了，环境也前进了，记忆更多了，任务难度也可能变化。较弱模型甚至可能因无法利用日益复杂的脚手架而低于简单基线。论文还在边界上交替使用过程奖励、教师重标和软监督微调，使参数 $\theta$ 与 Harness 状态 $H$ 以不同时间尺度变化。

[定位：H2 4.5，第 1—8 段]

4.6 Harness 演化与 Agent 自指的区别

AutoHarness、Meta-Harness、AHE、Self-Harness、RHO 与 Continual Harness 分别占据不同坐标。某系统可以拥有很宽的 Harness 编辑面却使用固定更新器，也可以编辑面较窄但更新者具有自指性。“Harness 演化”回答改动什么执行表面，“Agent 自指”回答谁构造谁的继任者。

无论哪种系统，治理边界都应保护权限上限、凭证、评价器完整性、资源限制、审计和回滚权。搜索拓扑与参数耦合还需另行说明。

[定位：H2 4.6，全文及六系统对照表]

5. 三项未来挑战

5.1 在没有真值时，究竟朝什么方向改进

现实任务常没有即时标签。自反思、自偏好、生成测试和替代验证器能提供搜索方向，但都有共同偏差、迎合评分器或降低标准的风险。关键设计原则是分离“提出修改所用的证据”与“批准修改上线的权力”。

作者建议建立可证伪的评价契约：什么证据会证明当前方向错误；哪些评价信号可以适应；哪些权力必须留在闭环之外；何时应弃权、延迟或请求人类复核。

[定位：H2 5.1，第 1—12 段]

5.2 从局部验证走向持久改进

一次 benchmark 上升可能来自过拟合、数据泄漏、增加成本、牺牲少数任务或暂时适应环境。真正的改进应在时间、任务、模型、用户和分布变化下保持价值，并计入延迟、费用、维护负担、安全回归与回滚频率。

评价单位最终应是性能轨迹而非单个分数：在有意义时间窗内，未来效用扣除成本、回归与维护后仍然提高。

[定位：H2 5.2，全文]

5.3 协调参数学习与外部可编辑状态

外部状态更新快、透明、可迁移和可回滚；参数学习推理时更紧凑，适合内化广泛稳定的模式。只依赖前者会导致脚手架膨胀、检索成本与冲突累积；过早把弱经验固化进参数，又会降低可解释性、带来灾难性遗忘，并改变现有 Harness 所适配的轨迹分布。

真正问题不是在“权重还是文件”之间二选一，而是分配与时间尺度：哪些经验先以可撤销外部状态试验，哪些在多次验证后压缩进参数，参数更新后哪些 Harness 假设必须重新验证。

[定位：H2 5.3，全文]

核心概念与方法

  • Persistent editable state：跨运行保存、能改变未来行为的模型外状态。
  • Updater–target relationship：谁根据证据构造谁的继任版本。
  • Evaluation and promotion：区别诊断、候选评价、档案保留与生产晋升。
  • Governance boundary：权限、凭证、预算、审计、sandbox、隐藏评价和回滚组成的外部信任根。
  • Search topology：决定改进是贪心替换、树搜索、Pareto、多谱系还是开放档案。
  • Skill lifecycle：发现、生成、验证、注册、检索、监控、修订、合并、弃用与回滚。
  • Reset-free adaptation：不重置环境和历史，在同一持续轨迹中修改 Harness。
  • Falsifiable evaluation contract：预先说明何种证据会否定改进方向，以及谁拥有最终裁决权。

局限与适用边界

  • 这是作者的统一视角，不是系统综述：没有说明检索协议、纳入排除标准或覆盖率，因此不能据此断言文献全集已经被覆盖。
  • 大量引用发表于 2026 年，成熟度、复现情况与同行评审状态各异；本文主要总结论文自报机制和结果。
  • 坐标分配与论文关系多为作者综合，不等同于原论文的自我定位。文章已主动标注这一证据边界，阅读时仍需保留。
  • $H$ 的边界在真实系统中可能模糊，例如工具代码、权限路由、测试和更新器可能位于同一仓库。概念分离不能替代实际 threat model。
  • “独立评价”有程度差异。同一模型不同上下文的 evaluator、隐藏测试、外部 oracle 和人工审批，独立性并不等价。
  • 文章提供分类框架而非统一实验；不能从跨论文对照直接推导方法优劣。
  • 全文强调持久性和评价，但对隐私、数据许可、恶意经验注入、跨用户传播和撤回机制仍只做原则性讨论。

对自改进 Agent Systems 主题的贡献

本文最有价值的贡献，是把容易混成营销标签的“自我改进”还原成可审计的系统工程问题。它给出一套可用于阅读论文、设计系统和做上线评审的问句：

  1. 哪些状态能被改，哪些状态明确不可改？
  2. 更新者与目标是什么关系？
  3. 修改依据来自标签、轨迹、测试、偏好、环境还是开放网络？
  4. 候选如何搜索、保存和淘汰？
  5. 修改何时生效，能否先验证再部署？
  6. 谁保护评价器、权限、预算、审计和回滚？
  7. 改进是否在未来任务和长期成本上仍成立？

这套框架也揭示一个很实用的结论：当前最可行的自改进往往不是模型修改自身权重，而是 agent 把经验沉淀成可 diff、可测试、可回滚的外部状态，再由受保护的评价与治理闭环决定是否保留。

可直接用于系统设计的检查表

  • 明确列出 $\theta$、$H$、$U$、$L$、$V$、$B$，禁止用“系统会自我学习”替代具体边界。
  • 所有更新以小型、可追踪差分表达，并保存来源、版本、触发证据和回滚点。
  • 将“提出候选”“进入实验档案”“灰度部署”“生产晋升”设计为不同状态。
  • 至少保留一条候选无法修改的评价路径，并对 evaluator drift 建立监控。
  • Skill 与 Harness 都建立回归集；验证不只覆盖目标任务，还覆盖成本、安全和旧能力。
  • 在线更新默认视为尚未证实的适应，直到后续轨迹显示净效用提升。
  • 跨用户共享 Skill 或 Harness 变更时，按软件供应链变更处理，而非普通记忆同步。
  • 在把经验蒸馏进参数前，先要求外部状态版本跨任务、跨时间稳定有效。

开放问题

  • 当人工标签稀缺时，怎样量化 evaluator 与 updater 的真正独立性？
  • 怎样设计能区分 Harness 修改、环境推进、记忆累积和任务难度变化的在线因果评价？
  • 开放档案中的弱分支应保留多久，怎样限制存储和搜索成本？

  • 哪类经验达到什么稳定性阈值后，才适合从外部状态固化进模型参数？
